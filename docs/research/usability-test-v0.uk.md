# Kachka · Юзабіліті-тест v0 — План і бланк фіксації

> Українська версія. Англійський оригінал: [usability-test-v0.md](usability-test-v0.md).
>
> Guerrilla юзабіліті-тест клікабельного прототипу v0 на друзях, що ходять у
> зал, **до** інвестиції у візуал. Легкий, solo, MVP.
> Читати разом із `../CONTEXT.md` і `../wireframes/INDEX.md`.

Статус: **готовий до запуску** · складено під прототип v0.

---

## 1. Мета

**Одна ціль:** переконатися, що ad-hoc модель тренування і ключові флоу
зрозумілі без пояснень — зловити структурні діри й діри розуміння поки правка
коштує абзац спеки, а не переробку візуальних макетів.

Це **не** візуальне/естетичне ревʼю (судити ще нема чого — `visual/` у статусі
v0) і **не** статистичне дослідження. Це маленький якісний прохід, щоб виявити
великі діри.

### Що цей прототип може й не може перевірити

Шар введення даних у прототипі статичний (див. `wireframes/INDEX.md` §«Два
режими»). Це жорстко обмежує скоуп:

| ✅ Можна перевірити зараз | ❌ Поза досяжністю цього прототипу |
|---|---|
| Ментальна модель: чи розуміють build → execute → log? | **Швидкість/ергономіку** логування сету (numpad статичний) |
| **Findability** — чи доходять до потрібного екрана під ціль? | Однією рукою, спітнілим пальцем, в умовах залу |
| Discoverability неочевидних жестів (тап по номеру сета, рядок `⋮`) | Реальний `±2.5/±5`, поведінку target ghost-text |
| Розуміння концепції **superset** | Введення тексту (назва, пошук вправ, custom create) |
| Навігацію між зонами (Today / History / Profile) | Поведінку toggle / filter-chip / поля пошуку |

> **Головний продуктовий ризик — in-gym мікро-ергономіка логування — цим
> тестом НЕ покривається.** Валідувати пізніше через Wizard-of-Oz (ведучий
> вручну перемикає екрани, коли учасник «тицяє») або на реальному білді. Тут
> полюємо лише на структурні дефекти й дефекти розуміння.

## 2. Метод

- **Тип:** task-based юзабіліті-тест, think-aloud, мовчазне спостереження.
- **Учасники:** 2–3 друзі, що регулярно тренуються. Трьох достатньо для
  guerrilla-проходу — ловить домінантні проблеми, не довгий хвіст.
- **Тривалість:** ~20–25 хв на людину.
- **Контроль bias:** друзі добрі. Мір те, що вони **роблять** (перший тап,
  заминки, хибні шляхи), не те, що **кажуть**, що їм подобається.

---

## 3. Сетап

1. Підніми папку wireframes сервером, щоб відносні `shared/` ресурси
   резолвились:
   ```bash
   # з D:\DEV\kachka\docs\wireframes
   npx serve .          # або: python -m http.server 8000
   ```
2. На телефоні учасника (та сама Wi-Fi) відкрий
   `http://<laptop-ip>:8000/prototype.html` — телефон у руці реалістичніший за
   десктоп-браузер.
3. З меню прототипу обирай стартовий сценарій, який називає завдання. Proto-
   режим (`?proto=1`) ховає анотації й показує тільки phone frame.
4. Записуй екран + аудіо, якщо учасник погодиться. Інакше ведучий заповнює
   бланк фіксації (§6) наживо.

## 4. Скрипт ведучого (читати вголос)

**Intro (anti-bias рамка):**

> «Дякую, що допомагаєш. Це чернетка дизайну додатка для тренувань — я
> перевіряю дизайн, а не тебе, помилитись тут неможливо. Найкорисніше для
> мене — якщо ти **думатимеш уголос**: що бачиш, що очікуєш, куди б тицьнув і
> чому. Я мовчатиму й просто дивитимусь — це нормально. Якщо щось бісить —
> кажи прямо, мені потрібна чесність, а не ввічливість.»

**На кожне завдання:** прочитай ціль, тоді мовчи. Якщо завис >~15 с:
> «Що ти зараз думаєш? Куди б тицьнув, якби мусив вгадати?»

Ніколи не показуй пальцем, не називай правильну кнопку, не рятуй зарано.

**Wrap-up:**
> «Що було найнезрозуміліше? Якби можна було змінити одне — що?»

## 5. Завдання

Відсортовані за ризиком — найризикованіші концепції першими, поки увага свіжа.
Прожени 5 core-завдань; додай опційні, якщо учасник залучений.

Легенда — **Success** = ціль досягнута без допомоги · **Partial** = досягнута
після заминки/повернення · **Fail** = хибний шлях або здався.

### T1 — Обрати правильний спосіб старту (CORE)
- **Старт:** `today-has-history.html`
- **Кажеш:** «Минулого разу ти робив тренування "Push A". Сьогодні хочеш
  зробити точно таке саме. Покажи, як.»
- **Правильний шлях:** тап **Repeat last**.
- **Success:** обирає *Repeat last*, а не *Choose from history* / *Start
  blank*, без вагань.
- **Лови:** Чи розуміють, що три опції різні? Чи «Repeat last» читається як
  «те саме тренування знову»? Плутанина між *Choose from history* і *Repeat
  last*.
- **Пробиває:** Today IA, основний флоу для юзера, що повертається (spec §3).

### T2 — Зібрати тренування, додати вправу (CORE)
- **Старт:** з T1 продовжити, або `today-has-history.html` → **Start blank
  workout** → `builder.html`.
- **Кажеш:** «Тепер збери тренування з нуля. Додай вправу Bench Press.»
- **Правильний шлях:** Quick-add chip *Bench Press*, **або** `+ Add exercise`
  → picker → тап по рядку.
- **Success:** вправа потрапляє у список будь-яким шляхом.
- **Лови:** Чи Quick-add chips читаються як «тапни, щоб додати»? Чи шукають
  замість цього пошук (поле пошуку статичне — познач як gap, не як finding)?
- **Пробиває:** модель Builder, discoverability Quick-add (spec §4).

### T3 — Створити superset (CORE · найризикованіша концепція)
- **Старт:** `builder.html` з ≥2 вправами (або стан row-menu через
  `builder-row-menu-sheet.html`).
- **Кажеш:** «Зроби так, щоб дві вправи виконувались по черзі як superset.»
- **Правильний шлях:** рядок `⋮` → **Add to superset** → config sheet.
- **Success:** відкриває superset config sheet.
- **Лови:** Чи знаходять рядок `⋮`? Чи розуміють «superset» так само, як його
  моделює додаток (по черзі, один rest)? Перший інстинкт (перетягнути дві
  разом? long-press?).
- **Пробиває:** найскладніша must-have концепція у v1 (spec §6). Якщо вона
  плутає людей — це найцінніший finding сесії.

### T4 — Позначити сет як warm-up (CORE · discoverability жесту)
- **Старт:** `in-workout.html`.
- **Кажеш:** «Твій другий підхід був розминочний. Познач його як warm-up.»
- **Правильний шлях:** тап по **номеру сета** → set-actions sheet → **Mark as
  warm-up**.
- **Success:** відкриває set-actions sheet через номер сета.
- **Лови:** Чи спадає взагалі на думку тапнути номер сета? (Відомий ризик —
  номер не виглядає тапабельним.) Де шукають першим ділом?
- **Пробиває:** affordance тапу по номеру сета (spec §8) — несуча, неочевидна
  взаємодія.

### T5 — Залогувати сет: лише ментальна модель (CORE · статична засторога)
- **Старт:** `in-workout.html`.
- **Кажеш:** «Ти щойно зробив 8 повторів на 60 кг. Покажи, куди б ти це
  вписав.»
- **Зауваж:** numpad статичний — ввести насправді не можна. Спостерігай, **де
  вони очікують вводити значення** і що очікують після тапу.
- **Success:** показують на поле ваги/повторів і артикулюють правильне
  очікування (зʼявиться numpad/quick-entry).
- **Лови:** Чи поля читаються як редаговані? Чи очікують системну клавіатуру
  проти кастомного пада? Чи шукають confirm/✓?
- **Пробиває:** ментальну модель логування (spec §5/§7). **Не** читай
  повільність як finding — тапи тут не реєструються.

### T6 — Завершити й зберегти (опційно)
- **Старт:** `in-workout.html`.
- **Кажеш:** «Тренування закінчилось. Заверши й збережи його.»
- **Правильний шлях:** Finish → completion screen → **Save**.
- **Лови:** Чи Finish findable? Чи completion screen сприймається як підсумок,
  а не зайва перешкода? Ясність Save проти Discard.

### T7 — Знайти, де додати власну вправу (опційно · findability)
- **Старт:** `profile.html`.
- **Кажеш:** «Ти робиш вправу, якої немає в додатку. Де б ти додав свою
  власну?»
- **Правильний шлях:** Profile → DATA → Exercise database → Create custom.
- **Лови:** Чи має сенс, що база вправ живе всередині Profile, чи очікують її
  деінде (напр. лише в picker)?

## 7. Відомі обмеження прототипу — НЕ записувати як findings

Це обмеження прототипу v0, не дизайн-дефекти. Якщо учасник застряг *через*
одне з них — пиши «заблоковано статичним прототипом», а не UX-проблему:

- Numpad / введення ваги / повторів — статичні, без набору.
- Поле пошуку вправ — статичне.
- Filter-chips муск-груп — статичні.
- Toggle налаштувань (Show RPE, haptics, theme, language) — статичні.
- Поля форми custom-вправи — статичні.
- Будь-який екран, не підключений до proto-лінків (клікабельну карту див. у
  `wireframes/INDEX.md`).

## 8. Синтез (після всіх сесій)

1. **Звести проблеми** по учасниках. Проблема, що зачепила ≥2 з 3 — сильний
   сигнал; одна драматична заминка теж важлива.
2. **Severity:**
   - **S1 Blocker** — не зміг виконати core-завдання (T1–T5).
   - **S2 Friction** — виконав, але з плутаниною/поверненнями.
   - **S3 Polish** — дрібне, косметичне або одноразове.
3. **Рішення на кожну:** фіксимо у `spec/<zone>.md` зараз, або логуємо в
   `spec/decisions.md` як свідоме відкладення. Привʼязка до §-секції.
4. **Гейт:** старт візуалу блокують лише S1. S2/S3 можуть їхати разом у
   візуальну фазу.

Додавай findings нижче як датований лог, щоб тест лишався живим записом.
