  • Tap на row → workout клонується per spec §3.2 → відкриває Workout Builder з заповненим списком (builder.html) з prev-значеннями взятими з обраного джерела (per spec §3.3)

Interactions

  • Vertical scroll з тими ж sticky section headers що у history-list.html
  • Swipe down від header — закриває modal (стандарт RN screen modals)
  • Tap row — клонування + push на Builder (modal стек заміщується або стак pushed; impl detail)
  • Без long-press / context menu — тільки tap-to-clone

Notes

  • Identical row rendering з history-list.html per task spec — намагалися не дублювати CSS логічно, але тут локальні стилі повторюються; у impl це буде один shared row component з prop mode: list | picker
  • Header difference: ✕ close замість title-only + центрований title Choose from history. Spacer справа (32px) щоб title залишався center-aligned (RN-style symmetric header)
  • Subheader hint Tap a workout to clone it — task spec вимагає; орієнтує юзера що тут tap-to-clone, а не tap-to-view (на відміну від list де tap → detail). Тон утилітарний, узгоджено з visual
  • Chevron справа залишений (як у list) — натяк на дію. Альтернатива — replace на іконку copy або + (підкреслить clone semantics). У v1 chevron достатньо, бо subheader hint вже дав контекст. Можливе уточнення
  • Без tab bar per task spec — це modal над Today (per spec §2.4 діаграма: Today --> CFH[Choose from history list])
  • Без infinite loader у цьому wireframe — не критично для picker mode (user скролить до знайомого workout-у і клонує). У impl infinite scroll буде той самий що в list
  • Per spec §3.3: prev-значення в клонованому workout-і беруться з обраного джерела, не з найостаннього такого workout-у. Свідомо — користувач обрав цей конкретний день як точку відліку

Edge cases · related states

  • Empty: якщо взагалі немає завершених workouts → Today показує empty state (single CTA per spec §3.1.b), entry point до picker не існує — отже empty state самого picker недосяжний у v1 flow
  • In-progress workout exists: tap Choose from history викликає prompt `Finish or discard current first` (per spec §2.2) — не доходить до picker. Open question (зафіксований у INDEX.md): що саме показуємо при такому tap — confirmation sheet або просто disabled lookup. TBD
  • Soft-deleted custom exercise у history workout-і → клонування зберігає назву і атрибути на момент кожного workout-а. Якщо вправа soft-deleted, при cloning вона лишається у списку (по entity ID); юзер може видалити її з Builder вручну
  • Workout з суперсетами: клонується вся структура (групи, rounds, rest) per spec §3.2 — у Builder з'являється з letter labels
